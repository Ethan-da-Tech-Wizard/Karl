# Risk Register — Karl v2

| Risk ID | Category | Description | Impact | Likelihood | Mitigation | Status |
|---|---|---|---|---|---|---|
| R01 | Threading | **GUI freeze on long inference.** `llama_cpp` runs a blocking C-loop. Calling it on the main thread causes "Not Responding." | Critical | High | All inference runs on `LLMThread` (QThread). Signals carry tokens back to UI thread. UI never blocks. | ✅ Resolved |
| R02 | Memory | **VRAM/RAM overflow.** 1.5B Q4_K_M needs ~1 GB RAM. Larger models spike higher during ingestion. OOM kills the process. | Critical | Medium | `hardware_scout.py` detects RAM/VRAM and gates model recommendations. `n_ctx` capped at 4096. User warned on upgrade. | ✅ Mitigated |
| R03 | Data | **Context window exhaustion.** Chat history grows until it exceeds the model's token limit, causing crashes or hallucination. | High | High | `LLMThread._trim_history()` implements a sliding window with per-message character cap and always preserves the seed message. | ✅ Resolved |
| R04 | User Error | **Catastrophic core modifications.** User edits `core/interaction_loop.py` with a syntax error or infinite loop. | High | High | All `core/` execution is wrapped in `try/except` in `LLMThread.run()`. Exceptions are formatted and emitted to the chat panel as red error messages. The app continues. | ✅ Resolved |
| R05 | Build | **`llama-cpp-python` compilation failure.** On Windows without C++ Build Tools, `pip install` fails silently or with cryptic errors. | Medium | High | README documents the exact pre-compiled wheel command. GPU path documented separately. | ✅ Documented |
| R06 | Data | **FAISS index corruption.** Sudden app closure during RAG ingestion leaves the `.faiss` and `metadata.json` out of sync. | Medium | Low | `save_index()` writes FAISS first, then metadata JSON. On load, both files must exist — if either is missing, the pipeline starts fresh. Partial writes are discarded. | ✅ Mitigated |
| R07 | Privacy | **Accidental telemetry.** `sentence-transformers` uses `huggingface_hub` internally, which attempts update checks and crash reporting. | High | High — was occurring | `main.py` sets `HF_HUB_OFFLINE=1`, `HF_HUB_DISABLE_TELEMETRY=1`, `HF_DATASETS_OFFLINE=1`, `TOKENIZERS_PARALLELISM=false` as the very first lines, before any import. | ✅ Resolved |
| R08 | Data | **Trace log grows unboundedly.** Daily JSONL trace files accumulate indefinitely. On high-use sessions, a single day's file can reach hundreds of MB. | Low | Medium | Currently unmitigated — `data/logs/` is gitignored. Future: add log rotation (keep last N days) to `TraceLogger`. | ⚠️ Open |
| R09 | Eval | **Grader false positives on empty output.** If the model produces an empty string, `json_valid` and `keyword_hit` may pass or error unexpectedly. | Medium | Low | `json_valid` fails on empty string (not parseable JSON). `keyword_hit` returns score 0.0 with all keywords as misses. `groundedness` returns FAIL for empty output. | ✅ Resolved |
| R10 | Training | **Logit bias applies wrong token IDs.** If the user switches models between sessions, cached logit bias token IDs may map to different tokens in the new model's vocabulary. | Medium | Low | `_parse_logit_bias()` re-tokenises at generation time using the currently loaded model instance — IDs are never cached across model switches. | ✅ Resolved |
| R11 | Training | **DPO export with no pairs.** User clicks Export DPO with zero corrected examples, resulting in an empty file with no error. | Low | Medium | `_export_dpo_data()` calls `get_dpo_stats()` first; if `dpo_pairs == 0`, shows an informational dialog explaining how to create pairs. Export dialog never opens. | ✅ Resolved |
| R12 | Training | **Training dependency pollution.** PyTorch, transformers, peft, and bitsandbytes installed into Karl's venv could conflict with llama-cpp-python or PyQt6. | Medium | Medium | Training dependencies are excluded from `requirements.txt` by design. `training/` directory is self-contained. `WHEN_TO_TUNE.md` instructs users to use a separate venv or Colab for training. | ✅ Mitigated |
| R13 | Eval | **EvalRunThread fails silently if no model is loaded.** Running an eval before loading a model raises an exception inside the thread. | Medium | Medium | `EvalRunThread` wraps all execution in `try/except`; `run_error` signal fires with the exception string, displayed in the runner panel's status log. | ✅ Resolved |
